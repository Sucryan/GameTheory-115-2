## Problem 1: Men-Proposing Deferred Acceptance

### 演算法邏輯

Problem 1 使用標準 men-proposing DA algorithm。

我的實作流程是：

1. 一開始所有 men 都是 free。
2. 每個 free man 依照自己的 preference list，向目前還沒提過的最高順位 woman proposal。
3. 如果 woman 目前沒有人，她就先 hold 這位 man。
4. 如果 woman 目前已經 hold 一位 man，她會比較新 proposer 和目前 held man。
5. woman 只保留自己比較喜歡的那一位，另一位重新變成 free。
6. 重複直到沒有 free man。

這個過程結束後，每個 man 都會被 match 到一位 woman，而且結果會是 men-optimal stable matching。

輸出為：

```text

0,22
1,25
...
```

## Problem 2: Stability Checker

### Blocking Pair 定義

對一組 matching 來說，如果存在一對 `(m, w)` 滿足：

1. man `m` 比起自己目前 matched woman，更喜歡 woman `w`。
2. woman `w` 比起自己目前 matched man，更喜歡 man `m`。

那 `(m, w)` 就是一組 blocking pair。只要有任何 blocking pair，這組 matching 就是不 stable。

### 實作方式

我先把 matching 轉成兩個方向：

```python
man_to_woman = {m: w}
woman_to_man = {w: m}
```

接著對每一個 man，只需要檢查他 preference list 中「排在目前 partner 前面」的 women。因為排在目前 partner 後面的 women，man 本人就不會想換過去，所以不可能形成 blocking pair。

概念上是：

```python
for man in men:
    current_woman = matching[man]
    for woman in men_prefs[man]:
        if woman == current_woman:
            break
        current_man = woman_to_man[woman]
        if women_rank[woman][man] < women_rank[woman][current_man]:
            blocking_pairs.append((man, woman))
```

最後輸出格式照助教 example：

```text
Matching_1: STABLE
Blocking Pairs: []

Matching_2: UNSTABLE
Blocking Pairs: [...]
```

我有注意到助教給的 `HW2_solution_example..csv` 比較像是輸出格式範例，而不是目前這組 data 的正確答案。因此這題沒有直接參考 example 的內容，而是重新根據 `HW2_m_prefs_10x10.csv`、`HW2_w_prefs_10x10.csv` 和 `HW2_matching_cases.csv` 計算 blocking pairs。

## Problem 3: Fairest Stable Matching

### Total Rank Sum

Problem 3 要找的是 stable matching 中 total rank sum 最小的那一個。

對每一對 matched pair `(m, w)`：

```text
score(m, w) = rank_m(w) + rank_w(m)
```

整個 matching 的 total rank sum 是所有 pair 的 score 加總：

```text
total_rank_sum = sum(score(m, w) for all matched pairs)
```

rank 是從 0 開始，所以如果某人 match 到第一志願，該側 score 就是 0。

### 為什麼不能只暴力枚舉

這題的資料是 12x12，如果直接枚舉所有 perfect matchings，數量是：

```text
12! = 479001600
```

這在 Python 裡雖然不是完全不可能，但非常不漂亮，而且沒有利用 stable matching 的結構。更重要的是，PDF 特別提到 non-brute-force algorithm 可以有 bonus，所以我不希望這題只寫成 naive permutation search。

### Optimization Strategy

我使用的是 **branch-and-bound + stability pruning**。

整體想法是：

1. 先跑一次 men-proposing DA 和 women-proposing DA。
2. 這兩個結果一定是 stable matching，所以可以先拿 rank sum 比較小的那一個當作 initial upper bound。
3. 接著用 DFS 建構 matching，每次固定一位 man 的 woman。
4. 在 DFS 過程中，如果 partial matching 已經不可能變成 stable，就直接剪枝。
5. 如果目前 partial matching 的 optimistic lower bound 已經不可能比 best answer 更好，也直接剪枝。

### Stability Pruning

DFS 還沒排完所有人時，有些 blocking pair 已經可以被確定，或是確定未來也無法修復。

我主要做三種檢查：

1. **已固定 pair 之間已經互相 blocking**
   - 如果兩邊都已經被 match，而且他們互相更喜歡彼此，那後面不管怎麼配都救不了。

2. **已固定 man 和未固定 woman 可能 blocking，但 woman 未來沒有更好的 remaining man 可以保護**
   - 如果 man `m` 比目前 partner 更喜歡 woman `w`，而 `w` 未來也不可能拿到比 `m` 更喜歡的 remaining man，那這個 partial matching 不可能 stable。

3. **已固定 woman 和未固定 man 可能 blocking，但 man 未來沒有更好的 unused woman 可以保護**
   - 這是第二點的對稱情況。

這些檢查的目的不是直接求答案，而是在 DFS 搜尋樹還很小的時候排除不可能 stable 的分支。

### Bound Pruning

除了 stability pruning，我還用了 rank sum 的 lower bound。

對於目前已經固定的 pairs，score 已經確定。對於還沒 match 的 men，我先樂觀地假設他們都能拿到目前 unused women 中 score 最低的那一位，得到一個 optimistic lower bound。

如果：

```text
lower_bound >= current_best_rank_sum
```

那代表這個分支即使在最樂觀狀況下，也不可能比目前答案更好，所以可以直接剪掉。

這邊使用 `>=` 的原因是作業只需要輸出 fairest matching 本身，不需要列出所有同分 matching。如果某個分支最多只能追平目前答案，就不用繼續展開。

## Problem 4: Many-to-One Matching with Quotas

### 題目設定

Problem 4 是 many-to-one stable matching。

- student 數量：50
- school 數量：5
- 每個 school quota：10

所以總 capacity 剛好是：

```text
5 * 10 = 50
```

每個 student 最後都會被分配到一間 school，每間 school 最多收 10 位 student。

### Student-Proposing DA

這題使用 student-proposing DA。

流程是：

1. 一開始所有 students 都是 free。
2. 每個 free student 向自己目前還沒提過、排名最高的 school proposal。
3. school 把新 proposer 加進目前 hold 的學生集合。
4. school 依照自己的 preference list 排序，只保留前 10 名。
5. 被擠出 quota 的學生重新變成 free，下一輪繼續向下一間 school proposal。
6. 重複直到沒有 free student。

實作上，每間 school 的暫存 list 會在加入新學生後排序：

```python
held_students[school].sort(key=lambda candidate: school_rank[school][candidate])
```

如果超過 quota，就把排名最差的學生 pop 出來：

```python
while len(held_students[school]) > quota:
    rejected = held_students[school].pop()
    free_students.append(rejected)
```

最後輸出格式為：

```text
school_id,student_id
0,2
0,24
...
```

## My Understanding

這次作業其實四題都圍繞在同一件事情：preferences 如何決定 stable outcome。

Problem 1 是最標準的 one-to-one DA，所以重點在於理解 proposal、hold、reject 這三件事。它的穩定性來自於 rejected proposer 往下一個選項走，而 receiver 永遠只保留目前看過最好的 proposer。

Problem 2 則是反過來檢查一個結果是不是 stable。這題讓我比較清楚看到 stable matching 的定義其實不是「大家都拿到很好的選項」，而是「不存在兩個人有動機私下離開原本 matching 去配對」。

Problem 3 比較像是在 stable matching 的集合裡再加上一個 fairness objective。DA 可以保證 stable，但不一定保證 total rank sum 最小，所以需要額外搜尋。不過如果完全暴力搜尋，又沒有利用 stable matching 的限制，因此我用 pruning 方式把不可能 stable 或不可能更好的分支提前排除。

Problem 4 則是把 DA 從一對一改成 many-to-one。直覺上 school 不再只 hold 一個人，而是 hold 一個大小最多為 quota 的集合。除此之外，核心精神其實仍然是 Deferred Acceptance：proposal 不是立即 final，而是暫時保留，直到後面可能被更好的 candidate 取代。
